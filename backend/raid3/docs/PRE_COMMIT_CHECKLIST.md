# Pre-Commit Checklist for raid3 Backend

This checklist aligns with the rclone [CONTRIBUTING.md](../../../CONTRIBUTING.md) guidelines. Use it before committing raid3 changes or opening a pull request for maintainer review.

---

## 1. Run Backend Unit and Integration Tests

**Why**: CONTRIBUTING requires running unit tests for changed code and, for backend updates, integration tests. raid3 has both in-package tests and framework tests that use remotes.

**What to do**:

- **From the rclone repository root**, run the raid3 package tests:
  ```bash
  go test ./backend/raid3 -parallel 1 -v
  ```
  Use `-parallel 1` to avoid test interference from shared state. Tests may take 1–2 minutes. Optionally run with `-race` to check for data races:
  ```bash
  go test ./backend/raid3 -race -parallel 1 -v
  ```

- **If you have a test config** (e.g. after `cd backend/raid3/test && ./setup.sh`), run the fs integration tests against the local raid3 remote:
  ```bash
  export RCLONE_CONFIG=backend/raid3/test/tests.config
  go test ./fs/operations -remote localraid3: -v
  go test ./fs/sync -remote localraid3: -v
  ```
  For MinIO (Docker required): use `-remote TestRaid3Minio:` and ensure MinIO containers are running (e.g. `./manage.sh start --storage-type=minio` from `backend/raid3/test`).

**Reference**: CONTRIBUTING § “Updating a backend”, § “Backend testing”; [TESTING.md](TESTING.md) in this directory.

---

## 2. Run Code Quality / Lint

**Why**: CONTRIBUTING states that installing golangci-lint and running the same checks as CI helps catch coding standard violations and common mistakes (e.g. unchecked errors).

**What to do**:

- From the **repository root**:
  ```bash
  make check
  ```
  or, if you have [golangci-lint](https://github.com/golangci/golangci-lint) installed:
  ```bash
  golangci-lint run ./...
  ```
  To limit to the raid3 backend:
  ```bash
  golangci-lint run ./backend/raid3/...
  ```

**Reference**: CONTRIBUTING § “Code quality tests”.

---

## 3. Documentation

**Why**: CONTRIBUTING asks for documentation to be added or updated for new or changed features. For raid3, that includes user-facing README, option help strings in code, and this docs folder.

**What to do**:

- **Links**: Ensure no markdown or comments link to non-existent files (e.g. removed `_analysis/` or implementation summary docs). Fix or remove broken references.
- **Option help**: If you added or changed backend options, keep the `Help:` text in the option struct (in `raid3.go`) as a single clear sentence; it is used for `rclone config` and generated docs.
- **User-facing changes**: If behavior or defaults change, update [../README.md](../README.md) and, if relevant, [../../../docs/content/raid3.md](../../../docs/content/raid3.md). Do **not** commit autogenerated doc output from `make backenddocs` unless you are doing a release; the PR check will flag changes in autogenerated sections.

**Reference**: CONTRIBUTING § “Writing documentation”, § “Backend documentation”.

---

## 4. Commit Message

**Why**: The first line of the commit message is used for the changelog and should be readable by users. CONTRIBUTING asks for the directory of the change and a colon, then a short summary.

**What to do**:

- **First line**: Prefix with `backend/raid3:` (or the specific sub-area, e.g. `backend/raid3/test:`) followed by a one-line summary of what changed from a user’s perspective. Example: `backend/raid3: fix broken doc links and clarify heal TODO`.
- **Body** (optional but recommended): After a blank line, add a short “why” and, if useful, before/after behavior. If the change fixes an issue, add `Fixes #1234` (or `#1234` to only link).

**Reference**: CONTRIBUTING § “Commit messages”.

---

## 5. Scope of the Commit / PR

**Why**: Keeping one logical change per commit (and a focused set of changes per PR) makes review and history easier.

**What to do**:

- For a single feature or fix (e.g. “fix doc links”), one commit is enough.
- For a larger contribution (e.g. the whole raid3 backend), one branch with a clear PR description is fine; avoid mixing unrelated refactors or other backends in the same PR so maintainers can review raid3 as a single feature.

---

## Quick Reference

| Step | Command / action |
|------|------------------|
| Backend tests | `go test ./backend/raid3 -parallel 1 -v` |
| Optional race | `go test ./backend/raid3 -race -parallel 1 -v` |
| fs integration (local) | `RCLONE_CONFIG=backend/raid3/test/tests.config go test ./fs/operations -remote localraid3: -v` (and same for `./fs/sync`) |
| Lint | `make check` or `golangci-lint run ./backend/raid3/...` |
| Commit | First line: `backend/raid3: <summary>`; body: why and, if needed, before/after |

See [TESTING.md](TESTING.md) for full test documentation and [../test/README.md](../test/README.md) for the Bash integration test harnesses.
